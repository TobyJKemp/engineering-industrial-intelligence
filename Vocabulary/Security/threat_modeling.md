# Threat Modeling

## One-Sentence Summary
Threat Modeling is a structured process of identifying potential security threats, attack vectors, and vulnerabilities in a system—enabling proactive security design rather than reactive patching.

## The Core Idea
- **Proactive:** Find threats before building
- **Systematic:** Use structured methodology
- **Attack thinking:** Think like attacker
- **Prioritization:** Focus on high-impact threats
- **Design guidance:** Informs security architecture

## How It Works
1. **System decomposition:** Break system into components
2. **Identify threats:** What could go wrong?
3. **Identify attackers:** Who might attack?
4. **Identify attack vectors:** How could they attack?
5. **Assess risk:** Which threats are highest risk?
6. **Mitigate:** Design controls for threats

## Think of It Like This
Threat modeling is like **disaster planning**: Before disaster strikes, identify potential disasters (earthquake, flood, fire), plan responses, build protections.

## The "So What?" Factor
- **Prevention:** Build security in from start
- **Prioritization:** Focus on real threats
- **Design:** Guides security architecture
- **Communication:** Common language for security discussions

## Watch Out For
⚠️ **Paralysis:** Threat modeling can be overwhelming.
⚠️ **Outdated:** Threats change; models need updating.

## Practical Checklist
- [ ] Have you identified all data flows in the system?
- [ ] Are threat actors and their motivations documented?
- [ ] Have you used structured methodology (STRIDE, PASTA)?
- [ ] Are threats prioritized by likelihood and impact?
- [ ] Are mitigation strategies assigned to threats?
- [ ] Have you documented security controls for each threat?
- [ ] Is threat model reviewed during design changes?
- [ ] Are team members trained on threat modeling?
- [ ] Have you tested security controls against modeled threats?
- [ ] Is threat model updated regularly (at least annually)?
