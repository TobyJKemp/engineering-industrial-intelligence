# Event Driven Architecture

## One-Sentence Summary
Event-driven architecture (EDA) is a design pattern where system components communicate through asynchronous event notifications—messages indicating something significant has occurred—rather than direct synchronous calls, enabling loosely-coupled systems where producers emit events (model training completed, data arrived, prediction threshold exceeded) without knowing who will consume them, and consumers react to events of interest without knowing who produced them, creating scalable, resilient architectures where components can be added, modified, or scaled independently while maintaining real-time responsiveness to changing conditions across distributed AI/ML systems.

## Why This Matters to You
You're building an ML platform where: model training takes 6 hours, generates artifacts that need validation, triggers deployment if validated, updates feature stores, notifies stakeholders, and logs metrics. With traditional synchronous architecture, training service must directly call validation service, wait for response, call deployment service, wait again, call feature store, wait again—each component tightly coupled, failures cascade, and if you add new downstream step (e.g., security scan), you must modify training service code. With event-driven architecture, training service emits single event: "ModelTrainingCompleted" with metadata (model_id, accuracy, artifacts_location). Validation, deployment, feature store, notification, and logging services all subscribe to this event, react independently and in parallel. Add security scan? New service subscribes to event—no changes to training service. This is why event-driven architecture matters—it transforms rigid, coupled workflows into flexible, parallel, scalable systems where components evolve independently. In AI/ML systems of 2026, EDA enables: real-time inference pipelines (events trigger immediate predictions), MLOps automation (pipeline stages triggered by completion events), data processing workflows (new data events trigger ETL), multi-model orchestration (model outputs as events feeding other models), monitoring and alerting (threshold events trigger automated responses), and A/B testing (user actions trigger variant selection). Studies show event-driven systems achieve: 10-100x better scalability (add consumers without burdening producers), 50-80% faster feature delivery (new capabilities added as event consumers without modifying existing services), 90% reduction in cascade failures (loose coupling prevents error propagation), and 60% lower latency for parallel workflows (consumers process simultaneously not sequentially). Without EDA: tight coupling (every service knows all downstream services), synchronous blocking (caller waits for all downstream processing), scalability bottlenecks (producers overwhelmed by consumer load), and rigidity (adding capabilities requires modifying existing code). With EDA: loose coupling (producers and consumers independent), asynchronous non-blocking (producers emit events and continue), independent scalability (scale consumers without affecting producers), and extensibility (add new consumers without changing producers). You might think "can't we just use API calls?"—but EDA provides: temporal decoupling (producer and consumer don't need to be available simultaneously), fanout (single event consumed by many services simultaneously), event history (replay past events for debugging or recovery), and backpressure management (consumers process at their own pace). Event-driven architecture is the foundation for resilient, scalable, evolvable distributed systems.

## The Core Idea
### What It Is
Event-driven architecture (EDA) is an architectural pattern where the flow of a program is determined by events—significant state changes or occurrences—rather than explicit control flow. In EDA, components are organized around production, detection, consumption, and reaction to events. The architecture emerged in the 1990s for GUI systems and financial trading platforms, gaining mainstream adoption in the 2000s with enterprise service buses, and becoming dominant in modern distributed systems with cloud-native architectures and real-time data processing.

The fundamental structure involves three core roles:

**Event Producers** - Components that detect state changes or occurrences and emit event notifications. Producers: generate events when significant things happen (model training completes, user clicks button, sensor reading exceeds threshold), package event data (what happened, when, relevant context), publish events to event infrastructure (broker, bus, stream), and continue processing without waiting for consumers. Producers are decoupled from consumers—they don't know who will consume events or even if anyone will. Example: ML training service completes training run, creates event `ModelTrainingCompleted(model_id="fraud-v2", accuracy=0.94, duration=6.2hours, artifacts_uri="s3://models/fraud-v2/")`, publishes to event broker, immediately starts next training run without waiting for downstream processing.

**Event Consumers** - Components that listen for specific events and react by executing business logic. Consumers: subscribe to event types of interest (ModelTrainingCompleted, DataArrived, PredictionThresholdExceeded), receive event notifications when they occur, process events asynchronously (validate model, update databases, trigger workflows), and may emit new events based on their processing (ValidationCompleted event after validating model). Consumers are independent—multiple consumers can process same event simultaneously, each performing different actions. Example: When ModelTrainingCompleted event arrives, validation service validates model quality, deployment service stages artifacts for deployment, metrics service logs training metrics, notification service emails stakeholders—all happening in parallel without coordination.

**Event Infrastructure** - The middleware connecting producers and consumers. Infrastructure: receives events from producers, stores events (temporarily or permanently), routes events to interested consumers based on subscriptions or filters, ensures reliable delivery (at-least-once, at-most-once, or exactly-once semantics), and provides event history/replay capabilities. Common infrastructure includes: message queues (RabbitMQ, AWS SQS), publish-subscribe systems (Apache Kafka, Google Pub/Sub, Azure Event Hubs), event buses (AWS EventBridge), and event streams (Apache Pulsar, Redis Streams). The infrastructure decouples producers from consumers—they communicate through events, not direct connections.

EDA operates through several key patterns:

**Publish-Subscribe (Pub-Sub)** - Producers publish events to topics/channels, consumers subscribe to topics of interest, infrastructure delivers events to all subscribers. Enables fanout—single event consumed by multiple independent services. Example: `ModelDeployed` event published to "deployments" topic, consumed by: monitoring service (set up dashboards), feature store (activate new features), documentation service (update API docs), audit service (log deployment), and rollback service (prepare rollback plan).

**Event Streaming** - Continuous flow of events captured in ordered, append-only logs (streams). Consumers process events in sequence, maintaining offsets (position in stream). Enables: event replay (reprocess old events), parallel processing (multiple consumers at different offsets), and event sourcing (stream as source of truth). Example: Click stream events flow continuously, multiple ML models consume stream at different rates—recommendation model processes in real-time, analytics model batches every hour, training pipeline samples 1% for retraining.

**Event Choreography** - Complex workflows emerge from services reacting to events without central coordinator. Each service: consumes events, performs its responsibility, emits new events. Workflow is implicit in event chains. Example: User checkout event → payment service processes payment and emits PaymentSucceeded → inventory service reserves items and emits ItemsReserved → shipping service creates label and emits ShippingLabelCreated → notification service sends confirmation email. No orchestrator—each service reacts to relevant events.

**Event Orchestration** - Central orchestrator manages workflow by: consuming events, maintaining workflow state, and commanding services (often through events). Orchestrator knows desired workflow sequence. Example: Order orchestrator receives OrderPlaced event, commands payment service (via PaymentRequested event), waits for PaymentSucceeded, commands inventory (via ReserveItems event), waits for ItemsReserved, commands shipping—orchestrator manages sequence.

In 2026 AI/ML systems, EDA enables critical capabilities:

**MLOps Pipeline Automation** - Model lifecycle stages triggered by events. Data ingestion triggers preprocessing, preprocessing completion triggers training, training completion triggers validation, validation success triggers deployment, deployment triggers monitoring setup. Each stage independent—swap preprocessing logic without affecting training, add security scanning as new consumer of training completion events. EDA makes pipelines: composable (mix-and-match stages), parallel (multiple models trained simultaneously from same data event), and extensible (add stages without modifying existing ones).

**Real-Time Inference Systems** - Prediction requests as events enable: load balancing (multiple inference services consume from event queue), request batching (accumulate events for efficient batch inference), fallback handling (if primary model service unavailable, secondary consumes events), and A/B testing (route percentage of events to variant models). Example: User action generates PredictionRequested event, routed to inference cluster where available service consumes event, performs inference, emits PredictionCompleted event with results—user-facing API subscribes to completion events for response.

**Multi-Agent Systems** - AI agents coordinate through events rather than direct calls. Agent completes task, emits TaskCompleted event with results. Other agents subscribe to events relevant to their responsibilities, react autonomously. Enables: dynamic agent graphs (agents join/leave without reconfiguration), parallel agent execution (multiple agents process same event simultaneously), and emergent behavior (complex workflows from simple event reactions). Example: Research agent emits FindingsDiscovered event, multiple specialist agents (statistical_analyst, visualizer, report_writer) consume event independently, each contributing their perspective, emitting their own events that trigger synthesis.

**Data Processing Pipelines** - New data arriving triggers cascade of processing. Raw data event triggers: validation (schema check), transformation (feature engineering), storage (data lake write), indexing (search index update), and notification (data catalog update). Each processing stage independent—scale transformation service without affecting validation, add new processing stage (e.g., quality scoring) as event consumer without modifying producers. Example: Sensor data event triggers: anomaly detection model inference, metrics aggregation, time-series database write, alert evaluation, and dashboard update—all in parallel.

**Model Monitoring and Alerting** - Predictions logged as events enable: drift detection (statistical monitoring service consumes prediction events, detects distribution shifts), performance tracking (actuals joined with predictions to compute metrics), A/B test analysis (compare prediction events from different model versions), and automated responses (drift detection service emits ModelDriftDetected event, triggering retraining pipeline). Monitoring is reactive—add new monitoring dimensions by adding event consumers.

**Feature Engineering Pipelines** - Raw events (user actions, transactions, sensor readings) flow through feature engineering services that: extract features, compute aggregations, enrich with context, and publish to feature store. Multiple feature pipelines can consume same raw events, each computing different feature sets. Enables: feature reuse (multiple models subscribe to same feature events), real-time features (features computed from streaming events), and independent feature evolution (add new feature pipeline without affecting existing ones).

The key architectural benefit: **loose temporal coupling**. Producers emit events whenever they occur, events persist in infrastructure, consumers process when ready—no requirement for simultaneous availability. If consumer crashes, restarts, processes events from last checkpoint—no data loss. This enables: resilience (consumer failures don't affect producers), independent deployment (update consumer without coordinating with producers), and time-shifted processing (batch consumer processes overnight events generated throughout day).

### What It Isn't
Event-driven architecture is not request-response architecture. Request-response is synchronous: client calls server, waits for response, proceeds based on response. EDA is asynchronous: producer emits event, continues immediately, consumer eventually processes. Confusion arises because some systems mix patterns—REST API receives request, translates to event, returns acknowledgment (hybrid). True EDA has no waiting—emit event and continue.

EDA isn't a replacement for all direct communication. Some interactions require synchronous responses: user query needs immediate answer (can't emit event and tell user "we'll get back to you"), transactional operations need immediate confirmation (did payment succeed?), interactive conversations require back-and-forth. EDA excels for: fire-and-forget operations (log this event), background processing (analyze this data when convenient), and fanout scenarios (notify everyone interested). Don't force EDA where synchronous interaction is appropriate.

Event-driven architecture doesn't eliminate the need for data consistency management. EDA typically operates with eventual consistency—consumer processes events asynchronously, there's delay between event production and all consumers completing processing. During this delay, system is in intermediate state. Some operations require strong consistency (financial transactions, inventory reservations)—EDA alone doesn't provide this. Must add patterns like sagas, two-phase commit, or distributed transactions. EDA makes eventual consistency more visible—design accordingly.

Finally, EDA isn't automatically more reliable than alternatives. Unreliable messaging creates problems: lost events (producer emits, infrastructure fails before persisting), duplicate events (infrastructure delivers same event multiple times due to retries), out-of-order events (events arrive in different order than emitted). Reliable EDA requires: durable event storage (events persisted before acknowledging producer), delivery guarantees (at-least-once or exactly-once), idempotent consumers (handling duplicate events safely), and ordering guarantees where needed (partitioning, sequence numbers). EDA provides building blocks for reliability—must use them correctly.

## How It Works
Implementing event-driven architecture systematically:

1. **Identify Event Types**: Determine what events your system needs. Analyze: state changes (model status changed, data arrived, user registered), significant occurrences (threshold exceeded, deadline passed, error occurred), and domain events (order placed, payment completed, product shipped). Name events clearly: use past tense (ModelTrainingCompleted not TrainModel), be specific (DataQualityCheckFailed not DataIssue), include context (PaymentSucceeded with payment_id, amount, method). Document event schema: what fields, data types, required vs optional. Example: Define events for ML pipeline: DataIngested, PreprocessingCompleted, TrainingStarted, TrainingCompleted, ValidationPassed, ValidationFailed, DeploymentSuccessful, DeploymentFailed.

2. **Choose Event Infrastructure**: Select technology for event transport and storage. For pub-sub with fanout: Apache Kafka (high throughput, durable, ordered within partitions), Google Cloud Pub/Sub (managed, scalable, simple), AWS EventBridge (AWS-native, schema registry, filtering). For message queues: RabbitMQ (complex routing, multiple protocols), AWS SQS (simple, managed, elastic), Azure Service Bus (sessions, transactions). For event streaming: Apache Kafka (industry standard for streams), Apache Pulsar (multi-tenancy, geo-replication), AWS Kinesis (AWS-native, simple). Consider: throughput requirements (events/second), durability needs (temporary queue vs permanent log), ordering requirements (global order vs partitioned order), delivery guarantees (at-least-once vs exactly-once), and operational complexity (managed vs self-hosted).

3. **Design Event Schema**: Establish event structure and evolution strategy. Common patterns: envelope (metadata + payload), CloudEvents standard (industry spec for event metadata), or custom schema. Include: event_type (what happened), event_id (unique identifier), timestamp (when occurred), source (what produced event), data (event-specific payload), and version (schema version for evolution). Use schema registries (Confluent Schema Registry, AWS Glue Schema Registry) for: versioning (track schema changes), validation (reject malformed events), and compatibility checking (ensure consumers can handle schema evolution). Example schema: `{event_type: "ModelTrainingCompleted", event_id: "uuid", timestamp: "2026-05-15T14:30:00Z", source: "training-service-v2", version: "1.0", data: {model_id: "fraud-v2", accuracy: 0.94, artifacts_uri: "s3://..."}}`.

4. **Implement Event Producers**: Build services that emit events. Producer code: detects significant occurrences (training completes, data arrives), constructs event object (populate schema fields), publishes to event infrastructure (call broker API), and handles publish failures (retry with backoff, dead letter queue for persistent failures). Best practices: publish after state changes are durable (write to database, then emit event—not reverse), include event causation (what triggered this event? useful for tracing), batch related events (multiple events from single operation), and ensure idempotent publishing (duplicate publishes don't cause issues). Example: Training service completes run, writes model artifacts to S3, writes metadata to database, then publishes ModelTrainingCompleted event—if publish fails, can retry safely since artifacts and metadata already persisted.

5. **Implement Event Consumers**: Build services that react to events. Consumer code: subscribes to event topics/queues, receives events (pull model: poll for events, push model: infrastructure invokes consumer), processes event (business logic), and acknowledges processing (tells infrastructure "I'm done, don't redeliver"). Handle failures: transient errors (retry), persistent errors (dead letter queue for manual investigation), and partial failures (saga pattern for compensating transactions). Ensure idempotency: processing same event multiple times produces same outcome (crucial since most systems guarantee at-least-once delivery). Example: Deployment service subscribes to ModelTrainingCompleted events, receives event, checks model artifacts exist, validates model format, stages to deployment directory, updates routing configuration, acknowledges event—if any step fails, event redelivered until success.

6. **Establish Delivery Guarantees**: Choose consistency level for event delivery. At-most-once: event delivered zero or one time (fast, simple, risk of loss—use for non-critical telemetry). At-least-once: event delivered one or more times (most common, guaranteed delivery, risk of duplicates—requires idempotent consumers). Exactly-once: event delivered exactly one time (complex, expensive, needed for financial transactions). Most systems use at-least-once with idempotent consumers (good balance). Implement idempotency through: deduplication keys (track processed event IDs), idempotent operations (set X=5 is idempotent, increment X is not), or unique constraint guards (database prevents duplicate inserts).

7. **Handle Event Ordering**: Determine ordering requirements. Global ordering: all events in exact emission order (simple but limits scalability—single partition bottleneck). Partitioned ordering: events within partition ordered (scalable—partition by entity_id so all events for given model ID are ordered). No ordering guarantees: events may arrive in any order (maximum scalability—but logic must handle out-of-order events). Most systems use partitioned ordering: partition key ensures related events are ordered (all events for model_id=X are ordered) while unrelated events process in parallel. Handle out-of-order events through: sequence numbers (track event order, detect gaps), timestamps (process based on event time not arrival time), and state reconciliation (handle late-arriving events updating past state).

8. **Implement Event Choreography or Orchestration**: Choose coordination pattern. Choreography (decentralized): each service reacts to events, emits new events, no central controller. Benefits: loose coupling, independent service evolution. Challenges: workflow understanding (implicit in event chains), debugging (trace event flows), circular dependencies (service A emits event causing B to emit event causing A again). Orchestration (centralized): workflow engine/service coordinates by: consuming events, maintaining workflow state, commanding next steps. Benefits: explicit workflow, easier debugging, timeout handling. Challenges: orchestrator as single point of failure, coupling to orchestrator. Choose choreography for: simple workflows (few steps), independent services (no strong sequencing). Choose orchestration for: complex workflows (many steps, conditional branching), transactional workflows (saga pattern coordination), timeout requirements (orchestrator tracks deadlines).

9. **Build Event Tracing and Observability**: Enable visibility into event flows. Implement: correlation IDs (track events belonging to same request flow—propagate through all events), event logging (record all emitted/consumed events), tracing (distributed tracing tools like Jaeger, Zipkin tracking event propagation), and visualization (tools showing event flows between services). Critical for: debugging (understand what happened), performance analysis (identify bottlenecks), and workflow understanding (see how services interact). Without observability, EDA is black box—events disappear into infrastructure, hard to understand system behavior.

10. **Implement Error Handling and Dead Letter Queues**: Handle processing failures gracefully. Retry policies: transient errors (network blip, downstream service temporarily unavailable) should retry with exponential backoff. Persistent errors: malformed event, business logic rejects event, downstream service permanently unavailable—after max retries, move to dead letter queue (DLQ). DLQ is: holding area for unprocssable events, enables manual investigation (why did processing fail?), allows fixing and reprocessing (fix consumer bug, replay DLQ events), prevents blocking (failed event doesn't block other events). Monitor DLQ size—growing DLQ indicates systemic issue. Example: Validation service receives ModelTrainingCompleted event, attempts validation, gets temporary S3 error (artifacts not available yet), retries 3 times with backoff, succeeds on retry. If artifacts are corrupt (permanent error), moves to DLQ after max retries for investigation.

11. **Manage Event Schema Evolution**: Handle schema changes without breaking consumers. Forward compatibility: new producer schema, old consumers—add optional fields only, don't remove/rename existing fields. Backward compatibility: old producer schema, new consumers—consumers handle missing fields gracefully. Use schema versioning: include version in event (version: "2.0"), consumers check version and handle appropriately. Schema registry enforces compatibility rules: prevent breaking changes, validate events match schema. Example: ModelTrainingCompleted v1.0 has accuracy field. V2.0 adds optional precision, recall, f1_score fields. Old consumers ignore new fields (forward compatible). New consumers provide defaults if fields missing (backward compatible).

12. **Implement Event Replay and Recovery**: Enable reprocessing past events. Event streaming systems (Kafka) retain events, consumers track offset (position in stream). To replay: consumer resets offset to earlier position, reprocesses events from that point. Use cases: fix consumer bug and reprocess (bug caused incorrect processing, fix bug, replay events to correct state), add new consumer (new service needs to build state from past events, replays history), and disaster recovery (rebuild system state from event log). Configure retention: balance storage cost vs replay capability. Critical systems keep events long-term (event sourcing—events are source of truth). Less critical systems keep days/weeks for bug recovery.

13. **Monitor Event System Health**: Track metrics for event infrastructure and processing. Producer metrics: events published per second, publish failures, publish latency. Infrastructure metrics: queue/topic depth (events waiting to be consumed—growing indicates consumers falling behind), throughput (events/second), disk usage (event storage). Consumer metrics: processing rate (events/second), processing latency (time from event emission to processing completion), error rate (failures per consumed event), lag (how far behind consumer is—critical metric). Alert on: growing lag (consumers not keeping up), high error rates (processing failing), failed publishes (events not reaching infrastructure). Example: Consumer lag grows from 100ms to 30 minutes—indicates consumer unable to keep up, need to scale consumers or optimize processing.

14. **Design for Backpressure and Flow Control**: Handle situations where consumers can't keep up with producers. Strategies: scaling (add more consumer instances), batching (process multiple events together for efficiency), throttling (slow down producers if consumers overwhelmed), priority queues (process critical events first), and filtering (only consume relevant events). Kafka's consumer groups provide automatic scaling: add consumer to group, partitions rebalanced, load distributed. Avoid cascading failures: if consumer falls behind, events pile up, infrastructure runs out of memory/disk, producers can't publish, entire system fails. Monitor consumer lag proactively.

## Think of It Like This
Imagine a restaurant kitchen. Traditional request-response is like: waiter takes order, walks to kitchen, waits while chef cooks, carries food back, serves customer. Waiter idle during cooking, can't take other orders, tightly coupled to chef's speed. Event-driven is like: waiter takes order, writes it on ticket, places on spindle (event broker), immediately takes next order. Chef pulls tickets from spindle when ready, cooks independently. When done, chef rings bell (emits event), food runner (event consumer) delivers. Multiple chefs can work in parallel, waiter doesn't wait, food runner can optimize delivery route—each role independent and scalable.

Adding capabilities is straightforward: restaurant wants to track popular dishes. Traditional approach: modify waiter process to log orders, modify chef to confirm dishes, modify runner to record deliveries—touch every component. Event-driven: add analyst who watches order tickets (subscribes to OrderPlaced events), records statistics, doesn't interfere with anyone—new capability added without modifying existing processes.

Kitchen coordinates through tickets and bells (events), not constant conversation. Chef doesn't need to know which waiter took order or which runner will deliver—just pulls ticket, cooks, rings bell. This decoupling enables: parallel work (multiple chefs on different orders), independent scaling (hire more chefs without more waiters), and specialization (different chefs for different cuisines—subscribe to relevant order types). Event-driven systems coordinate similarly—through event notifications, not direct coupling.

## The "So What?" Factor
**If you use event-driven architecture:**
- Loose coupling—components communicate through events not direct connections, evolve independently
- Scalability—add event consumers without burdening producers, scale components independently
- Parallel processing—multiple consumers process same event simultaneously, not sequentially
- Extensibility—add new capabilities as event consumers without modifying existing services
- Resilience—consumer failures don't affect producers, events persist until successfully processed
- Real-time responsiveness—react to events as they occur, trigger immediate actions
- Event history—replay past events for debugging, recovery, or building new consumer state
- Temporal decoupling—producers and consumers don't need simultaneous availability
- Fanout—single event triggers multiple independent actions across different services
- Workflow flexibility—change workflows by adding/removing event consumers not modifying orchestration code
- System observability—event logs provide audit trail of everything that happened
- Backpressure management—consumers process at their own pace, infrastructure buffers events

**If you don't:**
- Tight coupling—services know about all downstream dependencies, changes cascade through system
- Scalability bottlenecks—producers constrained by slowest consumer, synchronous calls limit throughput
- Sequential processing—downstream operations happen one-after-another not in parallel
- Rigidity—adding capabilities requires modifying existing services to call new endpoints
- Cascade failures—consumer failures propagate to producers causing widespread outages
- Batch-oriented—process accumulated data periodically not react to events in real-time
- No history—once processing complete, original request lost, can't replay or recover
- Temporal coupling—all components must be available simultaneously for processing to succeed
- Single consumer—one service handles each request, no parallel processing by multiple services
- Hardcoded workflows—workflow logic embedded in code, changes require code modifications
- Limited observability—interactions through direct calls harder to trace and audit
- Backpressure problems—fast producers overwhelm slow consumers causing crashes or dropped requests

## Watch Out For
⚠️ **Event Ordering Assumptions**: Assuming events arrive in emission order without ordering guarantees. Developer emits TrainingStarted then TrainingCompleted, assumes consumers see same order. But without ordering guarantees (or with multiple partitions), events can arrive out-of-order. Consumer sees TrainingCompleted before TrainingStarted—logic breaks. Solution: Don't assume ordering unless infrastructure provides it. Use partitioned ordering (partition by model_id ensures all events for given model ordered). Include sequence numbers in events to detect out-of-order arrival. Design logic to handle out-of-order events (state machines, version timestamps). Test with intentionally shuffled event order.

⚠️ **Ignoring Idempotency**: Consumers that aren't idempotent fail with at-least-once delivery. Consumer receives ModelTrainingCompleted event, increments metrics counter, acknowledges. Event redelivered due to network issue, counter incremented again—now incorrect. Problem: most event systems guarantee at-least-once (not exactly-once), so duplicates will happen. Solution: Make consumers idempotent—processing same event multiple times produces same outcome. Track processed event IDs (deduplication), use idempotent operations (set value, not increment), or database uniqueness constraints (prevent duplicate inserts). Test by processing same event multiple times, verify correctness.

⚠️ **Losing Causality and Tracing**: Events flow through system with no correlation, impossible to trace related events. ModelTrainingCompleted triggers validation which triggers deployment which triggers monitoring setup—four events, no connection, can't trace flow. Debugging requires manual correlation—"was this deployment caused by this training?" Solution: Implement correlation IDs—include causation_id in every event pointing to triggering event. Generate trace_id at system entry, propagate through all events. Use distributed tracing (Jaeger, Zipkin) to visualize event flows. Build tooling to query "show all events for trace_id=X" revealing complete flow.

⚠️ **Event Schema Chaos**: Each service emitting events with different structures, no standards. One service uses `timestamp`, another `event_time`, another `created_at`. Consumers must handle multiple formats. Adding fields inconsistently breaks consumers. Solution: Establish event schema standards—required fields (event_type, event_id, timestamp, source, data), naming conventions (snake_case, use standard field names), and versioning (include version field). Use schema registry (Confluent, Glue) to: enforce schemas, validate events, check compatibility. Reject non-conforming events at publish time.

⚠️ **Dead Letter Queue Neglect**: Failed events moved to dead letter queue (DLQ), then forgotten. DLQ grows to thousands of events, no investigation, underlying bugs never fixed. Lost data, no one notices. Solution: Monitor DLQ size—alert when growing. Establish DLQ triage process: investigate failures, fix root causes (consumer bugs, schema issues), replay fixed events. DLQ should be small and temporary—if large, indicates systemic problem. Set retention limits on DLQ (force investigation before events expire). Some teams review DLQ daily.

⚠️ **Eventual Consistency Surprises**: Forgetting EDA implies eventual consistency causes user-facing inconsistencies. User trains model, immediately queries model registry, model not yet shown (validation service hasn't processed TrainingCompleted event yet). User confused—"I just trained it!" From event perspective: training emitted event, registry will update eventually, but not instant. Solution: Design UIs for eventual consistency—show "Processing..." states, optimistic updates (show expected state before confirmation), or polling (refresh until appears). For critical operations needing immediate consistency, use synchronous calls (query training service directly) or saga patterns (coordination protocol). Don't expose eventual consistency gaps to users.

⚠️ **Event Explosion**: Emitting too many fine-grained events creates performance problems. Service emits event for every database row change—millions of events per second. Infrastructure overwhelmed, consumers can't keep up, costs explode. Solution: Choose appropriate event granularity. Emit events for business-significant occurrences (model trained, deployment completed), not technical details (row updated). Batch related events (single DataBatchProcessed with count=10000 rows, not 10000 RowProcessed events). Use Change Data Capture (CDC) carefully—filter to relevant changes. Test event volume under load.

⚠️ **Circular Event Dependencies**: Service A emits event triggering B, B emits event triggering A—infinite loop. Training service emits ModelUpdated, monitoring service consumes it and emits MetricsUpdated, training service consumes MetricsUpdated and emits ModelUpdated again—cycle continues until system crashes. Solution: Design event flows as directed acyclic graphs (DAGs), not cycles. Services should: consume events, perform work, emit completion events—but not emit events that trigger themselves. Review event choreography diagrams to detect cycles. Implement cycle detection (track causation chains, alert if circular). Sometimes cycles are intended (feedback loops)—then need termination conditions (max iterations, convergence checks).

⚠️ **Ignoring Consumer Lag**: Consumers falling behind producers but no alerts, lag grows from seconds to hours to days. Events pile up, infrastructure runs out of storage, producers can't publish. Problem discovered only when system fails completely. Solution: Monitor consumer lag continuously—difference between event production time and consumption time. Alert when lag exceeds thresholds (5 minutes, 1 hour). Investigate lag causes: slow consumer (optimize code), insufficient consumers (scale out), or event surge (temporary spike). Kafka provides consumer lag metrics out-of-box—use them. Consider lag budget (acceptable lag under normal conditions) and lag SLO (maximum acceptable lag).

## Quick Decision Guide
**Use event-driven architecture when:** Building distributed systems with multiple independent components, need loose coupling between services, require fanout (single event triggering multiple actions), need scalability (scale producers and consumers independently), have asynchronous workflows (background processing acceptable), require event audit trail (know what happened when), need temporal decoupling (producers and consumers available at different times), have evolving systems (frequently add new capabilities), require resilience (component failures shouldn't cascade)

**Simpler approaches sufficient for:** Simple monolithic applications (single process), synchronous request-response sufficient (immediate response needed), strong consistency required everywhere (no eventual consistency acceptable), few components (tight coupling manageable), simple linear workflows (no fanout or parallelism), low event volume (hundreds not millions per second)

**Event-driven architecture critical when:** Building MLOps pipelines (training → validation → deployment workflows), real-time inference systems (prediction requests as events), multi-agent AI systems (agents coordinate through events), data processing pipelines (streaming data transformations), monitoring and alerting (reacting to threshold events), workflow automation (business process automation), integration platforms (connecting many systems), IoT systems (sensor events triggering actions)
