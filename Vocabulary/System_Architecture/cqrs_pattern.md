# CQRS Pattern

## At a Glance
| | |
|---|---|
| **Category** | System Architecture Pattern |
| **Complexity** | Advanced |
| **Time to Learn** | Weeks to understand principles, months to implement effectively |
| **Prerequisites** | Event-driven architecture, database design, API design, distributed systems |

## One-Sentence Summary
CQRS (Command Query Responsibility Segregation) is an architectural pattern that separates write operations (commands that modify state) from read operations (queries that retrieve data) into distinct models, paths, and often separate data stores—enabling you to optimize each side independently, such as using a normalized relational database for writes to ensure consistency while maintaining denormalized read replicas optimized for fast queries, a separation particularly valuable in AI systems where model inference (read-heavy) has vastly different performance requirements than model updates and training data ingestion (write-heavy).

## Why This Matters to You
Your ML prediction service handles 10,000 inference requests per second (reads) but only updates the model once per day (writes). Using a traditional CRUD architecture with single database, you're forced to compromise: optimize for read performance (denormalized, cached, replicated) but make writes complex, or optimize for write consistency (normalized, transactional) but make reads slow. Neither is ideal. Every inference query hits the same database handling model updates, causing contention. Scaling requires scaling everything together even though 99.99% of traffic is reads. **This is why CQRS matters**—it allows you to optimize reads and writes independently because they have fundamentally different requirements, performance characteristics, and scaling needs. In AI systems of 2026, the read/write asymmetry is extreme: inference serving might handle millions of predictions per day (reads) while model registry updates happen dozens of times (writes), training data ingestion processes billions of records in batch (writes) while feature stores serve features for real-time inference at microsecond latency (reads), experiment tracking captures thousands of metrics during training (writes) but dashboards display aggregated results (reads). Forcing these through single unified model creates impossible trade-offs. CQRS recognizes that reads and writes are fundamentally different operations with different optimization strategies: writes need consistency, validation, transactions, and audit trails; reads need speed, denormalization, caching, and specialized indexes. By separating them, you can: use PostgreSQL for transactional writes with strong consistency, replicate to Elasticsearch for fast full-text search reads, cache frequently accessed data in Redis for microsecond latency, maintain specialized read projections for analytics, scale read and write infrastructure independently based on actual load. Studies show CQRS can improve read performance 10-100x through denormalization and caching while maintaining write consistency, reduce infrastructure costs 40-60% by scaling reads/writes independently, and enable architectural flexibility impossible with unified models. For AI systems specifically: model serving scales horizontally to handle prediction load while model management remains simple single-writer system, feature stores optimize reads for inference latency while feature engineering writes run in batch, experiment tracking captures all training metrics while dashboards query aggregated views. Without CQRS, these systems either compromise performance (slow reads or inconsistent writes) or accept massive complexity (maintaining consistency across optimized paths). CQRS is particularly critical for: event-sourced AI systems (capturing every decision for auditability), high-throughput inference services (millions of predictions with occasional model updates), real-time feature stores (microsecond reads, batch writes), and multi-tenant AI platforms (different read/write patterns per tenant).

## The Core Idea
### What It Is
CQRS (Command Query Responsibility Segregation) is an architectural pattern that fundamentally separates command processing (operations that change state) from query processing (operations that retrieve data), creating distinct models, paths, and often separate storage optimized for each concern. Introduced by Greg Young in 2010 and building on Bertrand Meyer's Command-Query Separation principle from the 1980s, CQRS has become increasingly relevant as systems grow more complex and read/write patterns diverge.

The core principle: **different operations have different requirements**. Commands (writes) need: strong consistency (changes appear atomically), validation (ensure business rules), transactions (all-or-nothing semantics), audit trails (who changed what when), and durability (don't lose changes). Queries (reads) need: low latency (return results fast), denormalization (avoid expensive joins), caching (serve repeated queries quickly), specialized indexes (optimize specific query patterns), and eventual consistency (slight staleness acceptable if fast). Traditional CRUD architecture forces single model serving both needs—impossible to optimize both simultaneously.

CQRS architecture consists of:

**Command Side (Write Model)** - Handles all state changes through commands. Commands are: intentions expressed as verbs (CreateModel, UpdatePredictionThreshold, IngestTrainingData), validated (business rules checked), processed (state changes applied), and recorded (events emitted or changes persisted). Command model optimized for: write consistency (transactions ensure atomicity), validation logic (complex business rules), strong typing (prevent invalid states), and audit trails (capture who/what/when). Example: model registry command side validates model submissions (correct format, required metadata, unique name), persists to PostgreSQL with foreign key constraints ensuring referential integrity, and emits ModelCreated event for downstream processing.

**Query Side (Read Model)** - Handles all data retrieval through queries. Queries are: questions (GetModelByName, ListProductionModels, SearchExperiments), optimized for specific access patterns (denormalized, pre-computed, indexed), eventually consistent (may lag command side slightly), and specialized (multiple read models for different needs). Query model optimized for: read performance (denormalized to avoid joins), specific queries (indexes and schemas tailored to common patterns), caching (frequently accessed data cached), and scalability (read replicas, CDNs). Example: model registry query side maintains: Elasticsearch index for full-text search (fast "find models by keyword"), Redis cache for frequently accessed models (microsecond lookups), and denormalized view with precomputed metrics (avoid aggregating at query time).

**Command Handlers** - Process commands, enforce business logic, and update write model. Responsibilities: validate commands (check preconditions, enforce rules), apply state changes (update database transactionally), emit events (notify about changes), and handle errors (provide meaningful feedback). Command handlers are: focused (one command type each), transactional (changes atomic), and idempotent (processing command twice has same effect as once). Example: UpdateModelThresholdHandler validates new threshold is in valid range (0-1), checks user has permission, updates PostgreSQL, and emits ThresholdUpdated event.

**Query Handlers** - Execute queries against read model. Responsibilities: fetch data from read stores (databases, caches, search indexes), apply filtering and pagination (handle query parameters), shape responses (transform to API contracts), and optimize performance (use caching, connection pooling). Query handlers are: stateless (no side effects), cacheable (same query returns same result until underlying data changes), and specialized (optimized for specific access patterns). Example: SearchModelsHandler queries Elasticsearch with filters, returns paginated results with highlight snippets, and caches results in Redis for repeated queries.

**Synchronization Mechanism** - Keeps read model eventually consistent with write model. Approaches: event-driven (write side emits events, read side consumes and updates), replication (database-level replication from write to read stores), batch (periodic jobs update read model from write model), or hybrid (combination of approaches). Synchronization ensures: read model reflects write model state (with acceptable lag), consistency (events processed in order), reliability (retries on failures), and scalability (async processing doesn't block writes). Example: When ModelCreated event emitted, EventHandler updates Elasticsearch index (for search), Redis cache (invalidates old cached lists), and analytics database (for reporting).

**Event Store (Optional but Common)** - Immutable log of all state changes as events. Events are: facts about what happened (ModelCreated, TrainingStarted, InferenceCompleted), immutable (never changed once written), ordered (timestamp and sequence), and complete (all information needed to reconstruct state). Event store enables: audit trails (complete history of changes), debugging (replay events to understand what happened), analytics (mine events for insights), and read model rebuilding (reconstruct read models from events). Example: Every model update recorded as event: {type: "ModelUpdated", modelId: "m-123", changes: {accuracy: 0.95}, timestamp: "2026-05-15T10:30:00Z", userId: "alice"}. Event log becomes source of truth.

In 2026, CQRS patterns are essential for AI systems:

**ML Model Serving** - Command side handles: model uploads (validate format and metadata), version registration (ensure unique versioning), and deployment requests (transition models to production). Write model in PostgreSQL ensures consistency. Query side handles: model lookups for inference (optimized for latency), model search (Elasticsearch for keyword/semantic search), and model comparison (precomputed metrics). Read models denormalized for fast access. Separation allows: scaling inference queries independently from model management, optimizing query stores for access patterns, and maintaining strong consistency for critical write operations.

**Feature Stores** - Command side handles: feature ingestion (batch and streaming writes), feature computation (transform raw data), and feature validation (quality checks). Writes optimized for throughput and consistency. Query side handles: feature retrieval for training (batch, high-volume), feature serving for inference (real-time, microsecond latency), and feature monitoring (analytics). Reads optimized for latency with: in-memory caching (Redis), precomputed features (avoid computation at query time), and specialized indexes. CQRS allows: batch writing features while serving reads with <1ms latency, maintaining consistent feature computation while optimizing serving, and scaling reads to millions of queries while writes remain manageable.

**Experiment Tracking** - Command side handles: experiment creation, metric logging (thousands of metrics during training), parameter updates, and status changes. Writes need strong consistency—can't lose experiment data. Query side handles: experiment search (find experiments by parameters), metric visualization (dashboard queries), comparison views (multi-experiment analytics), and leaderboard generation (aggregated rankings). Reads need: fast response (dashboard load times), complex aggregations (compare across experiments), and flexible filtering (parameter searches). CQRS enables: reliable write path ensuring no data loss, optimized read path with precomputed aggregations, and specialized read models (time-series database for metric plots, Elasticsearch for search, PostgreSQL for structured queries).

**AI Agent State Management** - Command side handles: agent actions (change world state), decision recording (what agent chose), policy updates (modify behavior), and learning (update models). Writes need: strong consistency (state changes atomic), validation (prevent invalid states), and audit (track agent decisions for debugging). Query side handles: current state retrieval (what's happening now), historical analysis (what did agent do), performance monitoring (aggregate metrics), and debugging (trace decision paths). Reads need: low latency (agent queries state frequently), flexible access (different debugging views), and analytics (understand behavior patterns). CQRS allows: guaranteeing consistent state transitions while enabling diverse analytical views of agent behavior.

**Multi-Tenant AI Platforms** - Command side handles: tenant-specific writes (model uploads, configuration changes, data ingestion) with isolation and validation. Query side handles: tenant-specific reads optimized per tenant's patterns. Some tenants inference-heavy (millions of predictions), others training-heavy (frequent retraining), others dashboard-heavy (analytics and monitoring). CQRS allows: isolated write paths with strong tenant boundaries, specialized read optimizations per tenant pattern, independent scaling (tenant with 10x inference load gets more read replicas without affecting others), and flexible storage (high-value tenants get dedicated read infrastructure).

The key insight: **reads and writes are fundamentally different**, and forcing them through unified model creates compromises that hurt both. CQRS acknowledges the difference and optimizes each independently. The trade-off is complexity—managing synchronization, handling eventual consistency, and maintaining multiple models. But for systems with significant read/write asymmetry or divergent optimization needs, the benefits outweigh costs.

### What It Isn't
CQRS is not simply separating read and write endpoints in your API. Many systems have distinct GET (read) and POST/PUT (write) endpoints but use the same underlying model and database—this is standard REST, not CQRS. True CQRS separates the models: different schemas, often different databases, potentially different consistency guarantees. The separation is deep, not superficial.

CQRS also isn't required for every system. It adds significant complexity: managing synchronization, handling eventual consistency, maintaining multiple models, debugging across paths. For systems with: modest scale (<1000 requests/second), similar read and write patterns, simple queries (no need for denormalization), or small team (complexity overhead too high), traditional CRUD is simpler and sufficient. CQRS pays off when read/write patterns diverge significantly—don't add complexity without clear benefit.

CQRS doesn't inherently mean microservices or distributed systems. You can implement CQRS in monolithic application with separate command and query services sharing same process. CQRS is about separating concerns, not distributing them. Though CQRS and microservices often combine naturally (separate services for commands and queries), they're independent patterns.

Finally, CQRS doesn't eliminate all data consistency challenges—it transforms them. Instead of ensuring single database stays consistent, you ensure read model eventually reflects write model. This is often easier (asynchronous, retryable) but requires: handling stale reads (query might return slightly outdated data), implementing idempotent updates (processing same event twice shouldn't corrupt state), and providing visibility (how far behind is read model?). CQRS trades immediate consistency for scalability and flexibility—not a free lunch but often a worthwhile trade.

## How It Works
Implementing CQRS effectively requires systematic approach:

1. **Identify Read/Write Asymmetry**: Analyze your system's access patterns. Measure: read vs write volume (10:1? 1000:1?), read vs write latency requirements (writes can be 100ms, reads must be <10ms?), query complexity (simple lookups vs complex aggregations?), and scaling needs (reads need 10x capacity?). CQRS justified when: significant read/write volume imbalance, different latency requirements, complex queries benefit from denormalization, or independent scaling valuable. If reads and writes are similar in volume/requirements, CQRS adds complexity without benefit.

2. **Design Write Model (Commands)**: Create command-focused model optimized for state changes. Define: command types (CreateModel, UpdateThreshold, DeleteExperiment—verbs expressing intentions), validation rules (business logic ensuring valid states), aggregate boundaries (what changes together transactionally), and state transitions (how commands modify state). Write model should be: normalized (avoid redundancy), consistent (enforce invariants), and transactional (changes atomic). Example: Model registry write model has: models table (id, name, version, created_by, metadata), deployments table (model_id, environment, status), foreign keys enforcing referential integrity, and constraints ensuring unique model names per version.

3. **Design Read Model (Queries)**: Create query-focused models optimized for specific access patterns. Identify: common queries (list production models, search by keyword, get model with metrics), performance requirements (latency targets), and access patterns (pagination, filtering, sorting). Read models should be: denormalized (precompute joins), indexed (optimize frequent queries), and specialized (different read models for different needs). Example: Model registry read models include: search index in Elasticsearch (full-text search), cache in Redis (frequently accessed models), and analytics view with precomputed metrics (avoid aggregating at query time).

4. **Implement Command Handlers**: Build handlers processing commands and updating write model. Handler responsibilities: validate command (check preconditions, enforce business rules), execute business logic (compute state changes), persist changes (update database transactionally), and emit events (notify about changes). Handlers should be: focused (one command type), transactional (atomic updates), idempotent (safe to retry), and instrumented (log actions for debugging). Example: CreateModelHandler validates metadata format, checks name uniqueness, inserts to PostgreSQL, emits ModelCreated event with model details, and returns success/failure with meaningful error messages.

5. **Implement Query Handlers**: Build handlers executing queries against read models. Handler responsibilities: parse query parameters (filters, pagination, sorting), execute query (against appropriate read store), format results (transform to API contract), and apply caching (avoid repeated expensive queries). Handlers should be: stateless (no side effects), optimized (use indexes, query efficiently), cacheable (leverage HTTP caching), and specialized (query appropriate read store). Example: SearchModelsHandler parses search terms, queries Elasticsearch with filters, returns paginated results with highlights, and sets cache headers for CDN caching.

6. **Choose Synchronization Strategy**: Decide how read models stay consistent with write model. Options: **Event-driven** (write handlers emit events, read handlers subscribe and update read models—most flexible and scalable), **Database replication** (PostgreSQL replicates to read replicas—simple but limited flexibility), **Change Data Capture** (CDC tools stream database changes to read stores—transparent to write logic), or **Batch synchronization** (periodic jobs update read models—simple but higher latency). For AI systems, event-driven typically best: flexible, scalable, enables multiple read models, and provides audit trail.

7. **Implement Event Publishing**: On write side, emit events for all state changes. Events should include: event type (ModelCreated, ExperimentStarted), relevant data (entity ID, changed fields, context), metadata (timestamp, user, causation ID), and be immutable (never modified after creation). Use: message queue (Kafka, RabbitMQ) for reliability, event schema versioning (handle evolution), and guaranteed delivery (retries, dead letter queues). Example: After creating model, publish: {type: "ModelCreated", modelId: "m-123", name: "fraud-detector-v2", version: "2.0", framework: "PyTorch", createdBy: "alice", timestamp: "2026-05-15T10:30:00Z"}.

8. **Implement Event Consumers**: On read side, consume events and update read models. Consumers should: subscribe to relevant events (filter by type), process events idempotently (safe to process twice), update read stores (Elasticsearch, Redis, analytics DB), handle failures (retry with exponential backoff), and maintain offsets (track last processed event). Use: consumer groups (parallel processing), checkpointing (resume after failures), and monitoring (track lag). Example: ModelCreatedConsumer receives event, indexes model in Elasticsearch for search, caches in Redis for fast lookup, and inserts into analytics DB for reporting.

9. **Handle Eventual Consistency**: Read models lag behind write model slightly (milliseconds to seconds). Provide: consistency guarantees documented (how long until reads reflect writes?), read-your-own-writes (user sees their changes immediately even if other reads lag), and visibility into lag (monitoring shows read model freshness). For critical consistency: query write model directly (slower but consistent), or return "processing" status (user knows change pending). For most cases: eventual consistency acceptable—most users won't notice sub-second lag.

10. **Optimize Read Paths**: Leverage CQRS separation to optimize queries. Strategies: **Denormalization** (flatten data to avoid joins—model details plus deployment info in single document), **Precomputation** (calculate aggregates during write—maintain running averages rather than computing on read), **Specialized indexes** (B-trees for range queries, full-text indexes for search, geo-indexes for location), **Caching** (Redis for hot data, CDN for static content), and **Read replicas** (scale horizontally—multiple Elasticsearch nodes, PostgreSQL read replicas). Measure query performance and optimize bottlenecks systematically.

11. **Ensure Write Consistency**: Write side must maintain strong consistency. Use: database transactions (all changes atomic or none), optimistic locking (detect concurrent modifications), aggregate boundaries (what changes together), and validation (enforce business rules before persisting). For distributed writes: consider saga pattern (coordinate across services), distributed transactions (two-phase commit when necessary), or eventual consistency (some writes can be async). AI systems typically benefit from: strong consistency for critical entities (model deployments, user accounts), eventual consistency for non-critical (usage metrics, logs).

12. **Implement Audit Trail**: Leverage event store or command log for auditability. Capture: all commands (who requested what), events (what actually happened), timestamps (when), and context (why—correlation IDs, user sessions). Audit trail enables: debugging (replay events to reproduce issues), compliance (prove what changes were made), analytics (understand usage patterns), and read model rebuilding (reconstruct from events). For AI systems: audit trails critical for: explaining model decisions (which model version made this prediction), debugging failures (what events led to this state), and compliance (demonstrate models were deployed properly).

13. **Monitor Synchronization Health**: Track read model freshness and synchronization. Metrics: event lag (time between write and read model update), processing rate (events per second), error rate (failed updates), and queue depth (pending events). Alert on: high lag (read models falling behind), high error rates (synchronization breaking), or stalled consumers (stopped processing). Provide visibility through: dashboards (real-time lag), health endpoints (service status), and logs (detailed processing traces). Healthy CQRS system has: low lag (sub-second for most), zero errors (or handled gracefully), and high throughput (processes events efficiently).

14. **Plan for Read Model Evolution**: As query requirements change, read models must evolve. Strategies: **Additive changes** (add new read models without modifying existing—safest), **Versioned models** (maintain multiple read model versions during migration), **Rebuild from events** (replay event store to construct new read model—expensive but thorough), or **Dual-write temporarily** (update both old and new models during transition). For large-scale systems: build tooling for read model management, automate rebuilding from events, and version APIs (clients can use new/old read models).

## Think of It Like This
Imagine a busy restaurant. The kitchen (write side) takes orders, prepares food, and updates order status. The dining room (read side) displays menu, shows table availability, and provides order status to customers. Using unified model would mean: every time customer checks order status, they'd interrupt kitchen asking "is my order ready?"—slowing food preparation. Every time hostess checks table availability, they'd search through kitchen orders—creating bottleneck. Kitchen optimized for cooking (consistency, proper sequencing), dining room optimized for customer service (fast access, pleasant presentation).

CQRS separates them naturally. Kitchen maintains order tickets (write model): detailed instructions, preparation status, allergies, modifications. Dining room maintains status board (read model): simplified order statuses, table numbers, estimated times. When kitchen completes order, they ring bell (emit event) and update status board (synchronize read model). Status board might lag seconds behind kitchen reality (eventual consistency) but that's acceptable—customers don't need microsecond accuracy on "your order will be ready in 8 minutes."

Separation allows optimization: kitchen uses format efficient for cooking (detailed tickets), dining room uses format efficient for customers (simple status board). Kitchen scales by adding cooks, dining room scales by adding displays. Kitchen ensures food consistency (proper cooking), dining room ensures customer experience (fast information).

AI systems work identically. Model registry write side (kitchen) ensures models are properly validated and consistently stored. Read side (dining room) provides fast model search, cached lookups, and analytics dashboards. Write side optimized for correctness, read side optimized for speed. They synchronize through events but operate independently—each optimized for its purpose.

## The "So What?" Factor
**If you implement CQRS:**
- Read performance improves 10-100x through denormalization, caching, and specialized indexes without compromising write consistency
- Write operations stay simple and consistent without compromise for read performance needs
- Independent scaling enables sizing read/write infrastructure based on actual load (not overprovisioning both)
- Multiple read models serve different access patterns without duplicating write logic
- Technology choices are flexible—best write database for consistency, best read stores for queries
- Eventual consistency is acceptable for most reads (microsecond-to-second staleness unnoticeable)
- Audit trails emerge naturally from event publishing—complete history of state changes
- Read models can be rebuilt from events without affecting write availability
- Development teams can work independently on read/write sides with clear contracts (events)
- Infrastructure costs drop 40-60% by avoiding overprovisioning and using appropriate storage per need
- Complex queries become simple through denormalization—no expensive joins at query time

**If you don't:**
- Performance compromises hurt both reads and writes—can't optimize both with single model
- Scaling is inefficient—must scale entire system even if only reads need capacity
- Complex queries require expensive joins or duplicate computation at query time
- Technology choices constrained—one database must serve all needs (write consistency AND read performance)
- Development bottlenecks occur—changes affect both read and write paths requiring coordination
- Infrastructure is overprovisioned—write capacity sized for read load or vice versa
- Analytics and reporting impact transactional performance—heavy queries slow writes
- No natural audit trail—must add logging separately across all state changes
- Read optimization complicates write model—denormalization violates normalization principles for writes
- System becomes brittle—high coupling between read and write logic creates fragility
- Evolution is difficult—optimizing for new read patterns requires write model changes

## Watch Out For
⚠️ **Premature CQRS**: Implementing CQRS for simple systems without clear read/write asymmetry. Team hears "CQRS is best practice" and adds complexity to application with 100 users handling 10 requests/second with balanced reads/writes. Result: massive complexity overhead (managing synchronization, eventual consistency, multiple models) with zero benefit. The development velocity tanks while maintaining dual models. Solution: Start with simple CRUD. Add CQRS when you have: clear read/write imbalance (>10:1), proven performance bottlenecks CQRS would address, and scale justifying complexity. Don't cargo-cult architecture patterns—apply when beneficial.

⚠️ **Ignoring Eventual Consistency**: Designing system assuming reads immediately reflect writes without planning for lag. User creates model, immediately searches for it, search returns empty—user confused. System races: write completed, event published, but read model not yet updated. Solution: Either accept eventual consistency and design UI accordingly (show "processing" message, provide feedback on write success), implement read-your-own-writes (user sees their changes immediately even if others don't), or query write model directly for critical consistency (slower but guaranteed fresh). Document consistency guarantees clearly.

⚠️ **Read Model Neglect**: Treating read models as second-class citizens—poor schema design, no optimization, inadequate monitoring. Read model is just "copy of write model with cache"—missing CQRS benefits. If read model doesn't serve queries significantly better than write model, why bother with complexity? Solution: Invest in read model design—denormalize thoughtfully, add specialized indexes, precompute aggregations, choose appropriate stores (Elasticsearch for search, Redis for caching, time-series DB for metrics). Read models are first-class design artifacts, not afterthoughts.

⚠️ **Synchronization Fragility**: Building fragile event processing that breaks on edge cases—duplicate events corrupt state, out-of-order events create inconsistencies, failures lose data. Read models become unreliable, requiring frequent manual rebuilds. Solution: Implement robust event processing—idempotent consumers (processing event twice doesn't corrupt), ordered processing (events processed in sequence), error handling (retries with exponential backoff), dead letter queues (capture unprocessable events), and monitoring (alert on high error rates). Synchronization must be production-grade, not prototype quality.

⚠️ **Command/Query Leakage**: Blurring boundaries between command and query sides. Commands doing reads to prepare responses (creating hidden coupling), or queries triggering side effects (violating read-only semantics). Impure separation loses CQRS benefits—reads and writes become entangled again. Solution: Maintain strict separation—commands modify state and return minimal response (success/failure, entity ID), queries read data without side effects. If command needs to return detailed entity, return ID and let client query separately. Separation discipline is critical.

⚠️ **Over-Engineered Event Store**: Building custom event sourcing infrastructure when simple event publishing suffices. Team implements full event store with event replay, projections, snapshots, temporal queries—massive engineering effort. Then discovers they only need audit trail and read model synchronization (much simpler). Solution: Start minimal—publish events to message queue, consume to update read models, store events for audit trail if needed. Add event sourcing complexity (replay, projections, snapshots) only if you need it (time travel, debugging complex state, regulatory requirements). Don't build event sourcing unless you're using event sourcing.

⚠️ **Ignoring Read Model Rebuilding**: No strategy for rebuilding read models when schemas change or data corrupts. Read model bugs create inconsistency, but only way to fix is manual database surgery. Or schema evolution requires complex migrations. Solution: Design for read model rebuilding from start—either maintain event log for replay (can reconstruct read models from events), or implement batch rebuilding from write model (query write database and reconstruct reads). Treat read models as derived data that can be rebuilt, not primary data that must be manually maintained.

⚠️ **Inadequate Monitoring**: Running CQRS without visibility into synchronization health. Read models lag behind by minutes but no one knows until users complain. Events fail silently, read models drift from reality. Solution: Comprehensive monitoring including event lag (time between write and read update), processing throughput (events per second), error rates (failed processing), queue depths (pending events), and consistency checks (periodic validation that read/write models match). Dashboard showing synchronization health in real-time. Alert on degradation before users notice.

⚠️ **No Versioning Strategy**: Evolving commands, events, or read models without versioning breaks existing consumers. Change event schema, old consumers crash trying to parse. Add required field to command, existing clients fail validation. Solution: Version everything—commands (v1/v2 APIs), events (schema versioning in messages), and read models (multiple versions during migration). Support backward compatibility, provide migration guides, and deprecate gracefully. In distributed systems, version changes require careful choreography.

## Quick Decision Guide
**Use CQRS when you have:** Significant read/write volume imbalance (>10:1), different performance requirements for reads vs writes, complex queries benefiting from denormalization, need for multiple read models optimized for different access patterns, independent scaling requirements, high-throughput inference with occasional model updates, real-time feature serving with batch feature computation, audit trail requirements for compliance

**Skip CQRS when you have:** Balanced read/write patterns, simple queries not benefiting from denormalization, small scale (<1000 req/s), tight consistency requirements (can't tolerate staleness), small team where complexity hurts velocity, simple CRUD applications, similar optimization needs for reads and writes

**CQRS critical for:** ML model serving (millions of inference reads, rare model updates), feature stores (microsecond reads, batch writes), experiment tracking (continuous metric writes, dashboard reads), AI agent platforms (heavy read traffic for monitoring with occasional commands), multi-tenant AI services (diverse per-tenant read/write patterns), event-sourced systems (need complete audit trail), high-scale applications (reads and writes need independent optimization)
